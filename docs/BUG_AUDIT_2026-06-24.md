# NSO Portal Bug Audit

Date: 2026-06-24  
Audited by: Codex  
Scope: frontend, backend, auth flow, data-loading paths, and build/lint health  
Constraint followed: no application code was modified

## Method

- Reviewed the React frontend, Express backend, and shared data-loading/auth paths.
- Ran frontend lint: `frontend/npm run lint`
- Ran frontend production build: `frontend/npm run build`
- Ran backend type-check: `backend/npx tsc -p tsconfig.json --noEmit`
- Cross-checked runtime risks against the current code paths and previously observed UI/runtime failures.

## Executive Summary

The codebase currently has a split authentication model:

- the frontend authenticates against Firebase/Auth + Firestore
- the backend still expects its own JWT token issued by `backend/src/routes/auth.ts`

That architectural split is the single highest-risk issue in the product. It causes feature breakage, inconsistent login/reset behavior, and makes some seeded backend users unusable from the current UI.

In addition, there are multiple data-classification bugs in store listing pages, several security defects in OTP/JWT handling, and a large volume of frontend lint/runtime-quality problems that increase the chance of page-level crashes.

## Confirmed Bugs

### 1) Frontend auth and backend auth are no longer compatible

- Severity: Critical
- Evidence:
  - `frontend/src/context/AuthContext.jsx` logs users in with `signInWithEmailAndPassword(...)`
  - `backend/src/routes/stores.ts` protects store routes with `authenticateToken`
  - `backend/src/routes/auth.ts` expects a backend-issued JWT
- What is wrong:
  - The current UI signs users into Firebase only.
  - The backend still expects its own JWT for protected API routes.
  - There is no bridge in the frontend that exchanges Firebase auth for the backend JWT.
- User impact:
  - Any backend feature that depends on JWT auth can fail even when the user is visibly “logged in”.
  - This creates inconsistent behavior between Firestore-backed pages and API-backed actions.

### 2) Backend token is read in the UI, but never written

- Severity: Critical
- Evidence:
  - `frontend/src/pages/AggregatorMail.jsx:1370` reads `localStorage.getItem('token')`
  - no matching `localStorage.setItem('token', ...)` exists in the frontend
- What is wrong:
  - The app tries to call JWT-protected backend routes using a token stored in local storage.
  - The current login flow never stores that token.
- User impact:
  - Template downloads and any similar backend action can send `token=null` and fail authorization.

### 3) Seed admin in the backend is not the same as a valid frontend login user

- Severity: Critical
- Evidence:
  - `backend/src/index.ts:55` seeds `admin@bluetokaicoffee.com`
  - `frontend/src/context/AuthContext.jsx` authenticates only through Firebase/Auth
- What is wrong:
  - A user seeded in Prisma/backend does not automatically exist in Firebase Authentication.
  - The login UI therefore cannot authenticate the seeded backend admin unless that user also exists in Firebase/Auth.
- User impact:
  - “Seed admin” credentials can appear valid in documentation or DB seeding, but still fail on the live login page.

### 4) OTP/email-link sign-in depends on local browser storage

- Severity: High
- Evidence:
  - `frontend/src/context/AuthContext.jsx` stores the email in `localStorage` under `nso_portal_email_link`
  - the email-link completion path refuses to proceed if that stored email is missing
- What is wrong:
  - The sign-in link flow only works reliably in the same browser profile that requested the link.
  - Opening the email on another device, browser, or after clearing storage breaks the login flow.
- User impact:
  - Users can receive a valid login link and still be unable to sign in.

### 5) Password reset architecture is inconsistent

- Severity: High
- Evidence:
  - `frontend/src/context/AuthContext.jsx:297` uses `sendPasswordResetEmail(...)`
  - `backend/src/routes/auth.ts:304` and `backend/src/routes/auth.ts:336` implement backend OTP reset endpoints
- What is wrong:
  - The frontend reset flow uses Firebase email reset.
  - The backend implements a separate OTP-based reset system.
  - These two systems are not unified.
- User impact:
  - Reset expectations are inconsistent.
  - Debugging mail/reset failures becomes unnecessarily difficult.
  - Product behavior differs depending on which path a user or developer assumes is active.

### 6) OTP reset API can report success even if email delivery fails

- Severity: High
- Evidence:
  - `backend/src/routes/auth.ts:318` calls `sendOTPEmail(user.email, otp);` without `await`
  - `sendOTPEmail(...)` catches its own send failures and only logs them
- What is wrong:
  - The route returns success before confirming that the message was actually delivered.
  - Delivery failures are swallowed inside the helper.
- User impact:
  - Users can be told that an OTP was sent when no email was actually delivered.

### 7) OTP values are exposed in logs and API responses

- Severity: Critical
- Evidence:
  - `backend/src/routes/auth.ts:319` logs `[OTP DEBUG] ... ${otp}`
  - `backend/src/routes/auth.ts:326` returns `otp` in the API response during local-testing mode
- What is wrong:
  - Secrets used for password reset are being exposed to logs and sometimes to API callers.
- User impact:
  - This is a direct credential-reset security flaw.

### 8) JWT secret has an insecure hardcoded fallback

- Severity: Critical
- Evidence:
  - `backend/src/routes/auth.ts:66` sets `JWT_SECRET = process.env.JWT_SECRET || 'bluetokai_jwt_secret_key_2026'`
- What is wrong:
  - If environment configuration is missing, the app falls back to a known static secret.
- User impact:
  - JWT signing becomes predictable and insecure in misconfigured environments.

### 9) Login domain handling is inconsistent for `@gottea.in` users

- Severity: High
- Evidence:
  - `frontend/src/pages/Login.jsx:29` fixes login initials to `@bluetokaicoffee.com`
  - `frontend/src/pages/Login.jsx:156` and `frontend/src/pages/Login.jsx:189` still allow `@gottea.in` for forgot-password and registration flows
- What is wrong:
  - The product accepts `@gottea.in` in some user journeys but not in the initials-based login input.
- User impact:
  - Got Tea users can register or request reset, but may be blocked from normal login entry.

### 10) “All Stores” silently hides inactive stores

- Severity: High
- Evidence:
  - `frontend/src/pages/Stores.jsx:135` runs `result = result.filter(s => s.isActive !== false);`
- What is wrong:
  - The page named “All Stores” excludes inactive stores before the user applies any filter.
- User impact:
  - Closed/inactive stores disappear from the supposed master list.
  - Reporting and manual verification from this page become incomplete.

### 11) “Upcoming Stores” includes records that are not truly upcoming

- Severity: High
- Evidence:
  - `frontend/src/pages/UpcomingStores.jsx:40-45` treats every store that is not `LIVE` or `CLOSED` as upcoming
- What is wrong:
  - Draft, on-hold, incomplete, rejected, or other workflow states are all grouped into “upcoming”.
- User impact:
  - The upcoming pipeline view is semantically inaccurate.
  - Decision-makers can misread the actual launch pipeline.

### 12) Store fetches can fail on slower auth/session restoration

- Severity: Medium
- Evidence:
  - `frontend/src/services/storeService.js:21` times out after 10 seconds waiting for auth state
- What is wrong:
  - Store reads fail if auth restoration is delayed or if auth has not fully initialized when the page loads.
- User impact:
  - Dashboard/store pages can intermittently fail or fallback even for valid users.

### 13) Production debug logging is still enabled across auth and data paths

- Severity: Medium
- Evidence:
  - `frontend/src/context/AuthContext.jsx`
  - `frontend/src/services/storeService.js`
  - `frontend/src/pages/Login.jsx`
  - `frontend/src/pages/Dashboard.jsx`
- What is wrong:
  - Internal auth state, user emails, and operational events are logged in production code.
- User impact:
  - Sensitive operational context is exposed in browser/dev logs.
  - Debug noise makes real incidents harder to isolate.

### 14) Production bundle is oversized

- Severity: Medium
- Evidence:
  - `frontend/npm run build` completed with a Vite warning
  - built JS bundle size: `dist/assets/index-p6EhY5LH.js` = `1,926.66 kB`
- What is wrong:
  - The main frontend bundle is too large and not sufficiently split.
- User impact:
  - Slower initial page load, worse performance on lower-end devices, and higher network cost.

## Confirmed Engineering Defects Likely to Surface as Runtime Bugs

These are not all proven from a live click path in this audit, but they are concrete code defects and should be treated as bug backlog items.

### 15) Frontend lint health is poor

- Severity: High
- Evidence:
  - `frontend/npm run lint` reports `90 problems (86 errors, 4 warnings)`
- Why it matters:
  - This volume of lint failures significantly raises the chance of hidden runtime defects and regression risk.

### 16) `ComplianceDetails.jsx` has a use-before-definition defect

- Severity: High
- Evidence:
  - lint flagged `fetchStoreDetails` as accessed before declaration in `frontend/src/pages/ComplianceDetails.jsx`
- Why it matters:
  - This is a concrete code smell that can become a runtime or refactor regression issue.

### 17) Multiple pages are doing state-setting inside effects in ways already flagged by lint

- Severity: Medium
- Evidence:
  - `frontend/src/pages/AggregatorMail.jsx`
  - `frontend/src/pages/ComplianceDetails.jsx`
  - `frontend/src/pages/Stores.jsx`
  - `frontend/src/pages/UpcomingStores.jsx`
  - `frontend/src/pages/UserRegistrations.jsx`
- Why it matters:
  - These patterns commonly cause unnecessary renders, stale state, and hard-to-debug UI behavior.

## Risks From Current Permissions Handling

### 18) Permissions are treated inconsistently as both strings and arrays

- Severity: Medium
- Evidence:
  - `frontend/src/context/AuthContext.jsx` defaults `permissions: ''`
  - some checks use string operations such as `.includes(...)`
  - other checks split the string with `.split(',')`
- What is wrong:
  - Permission logic is not normalized to one data type.
- User impact:
  - Access control can become inconsistent across pages and API routes.

## Recommended Priority Order

1. Fix the auth architecture split between Firebase/Auth and backend JWT auth.
2. Fix OTP/reset delivery behavior and remove secret exposure.
3. Repair login compatibility for seeded/admin users and `@gottea.in` users.
4. Fix store classification bugs in `Stores.jsx` and `UpcomingStores.jsx`.
5. Reduce lint/runtime defects in high-traffic pages.
6. Reduce bundle size and remove production debug logging.
